# Refactor Mode

進入重構模式，安全地改善程式碼結構而不改變功能。

## 使用方式

```
/refactor [檔案路徑或功能描述]
```

## 執行步驟

### 1. 分析目標

先理解要重構的程式碼：
- 讀取目標檔案
- 理解其功能和依賴關係
- 識別現有的測試（如果有）

### 2. 識別重構機會

檢查以下項目：

**結構問題**：
- [ ] 檔案過大（> 500 行）→ 考慮拆分
- [ ] 元件過大（> 200 行）→ 考慮拆分
- [ ] 函數過長（> 30 行）→ 考慮拆分
- [ ] 參數過多（> 5 個）→ 考慮使用物件參數

**設計問題**：
- [ ] 緊密耦合 → 使用依賴注入
- [ ] 重複邏輯 → 抽取共用函數
- [ ] 不一致的命名 → 統一命名慣例
- [ ] 缺少型別 → 補充 TypeScript 型別

**專案特定**：
- [ ] 使用 `inventory` 而非 `stock ?? inventory` fallback
- [ ] 使用 `storage_url` 而非 `storageUrl`
- [ ] 共用型別放在 apps/ 而非 packages/types/

### 3. 制定重構計畫

```markdown
## 重構計畫

### 目標
[一句話描述重構目標]

### 變更範圍
1. [檔案1] - [變更說明]
2. [檔案2] - [變更說明]

### 風險評估
- 影響範圍：[低/中/高]
- 破壞性變更：[是/否]
```

### 4. 漸進式執行

**重要**：每個小步驟都要驗證！

```bash
# 每次變更後執行
pnpm type-check
```

### 5. 最終驗證

```bash
pnpm type-check
pnpm build
```

## 重構安全守則

| 守則 | 說明 |
|------|------|
| 小步快跑 | 每次只做一個小變更 |
| 持續驗證 | 每步驟後都執行 type-check |
| 不改功能 | 重構不應改變外部行為 |
| 可回滾 | 確保每個 commit 都可獨立回滾 |

## 輸出格式

```markdown
# 重構報告

## 概述
[重構了什麼、為什麼]

## 變更清單
1. [檔案] - [變更描述]

## 驗證結果
- 型別檢查：[通過]
- 建置：[通過]

## 後續建議
- [如果有的話]
```
